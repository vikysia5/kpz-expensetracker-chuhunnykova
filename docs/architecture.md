# Архітектура проєкту Expense Tracker

## Опис домену
Система призначена для обліку особистих та спільних витрат користувачів, а також для аналізу фінансових даних.

### Яку проблему вирішує система?
Користувачам складно контролювати свої витрати та аналізувати фінансові звички. Система дозволяє зручно зберігати витрати, переглядати статистику та керувати бюджетом.

### Користувачі системи
- Гість — переглядає інформацію про застосунок
- Користувач — додає та аналізує витрати
- Спільний користувач — працює зі спільним бюджетом

### Основні операції
- Реєстрація та авторизація
- Додавання витрат
- Редагування та видалення витрат
- Перегляд статистики
- Розподіл витрат по категоріях
- Спільний доступ до витрат
- Експорт звітів

### Зовнішні системи
- База даних (MongoDB/PostgreSQL)
- Telegram API (для інтеграції)

## Рішення щодо якості коду

### Використані принципи SOLID

У проєкті застосовано принципи SOLID:

- SRP — кожен сервіс має одну відповідальність (TransactionService, BudgetService)
- OCP — новий функціонал додається через нові класи
- LSP — використання абстракцій дозволяє підмінювати реалізації
- ISP — інтерфейси розділені (Repository, Notification)
- DIP — залежності передаються через конструктор


### Структура модулів

- handlers/ — взаємодія з Telegram (Presentation)
- services/ — бізнес-логіка

### Яких Code Smells уникнуто

- Duplicate Code — логіка не дублюється
- Long Method — методи короткі
- Magic Numbers — всі значення винесені в константи
- Large Class — класи розділені
- Shotgun Surgery — зміни локалізовані


### Додаткові принципи

- DRY — немає дублювання
- KISS — прості рішення
- YAGNI — тільки потрібний функціонал

## Модуль реєстрації користувача
Відповідає за перегляд вступного екрану та створення нового акаунту з нікнеймом і паролем.

## Модуль витрат
Відповідає за додавання, збереження та обробку витрат користувача.
