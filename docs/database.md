# Database Documentation – Expense Tracker

## Загальний опис
База даних призначена для зберігання інформації про користувачів, їх фінансові операції (витрати/доходи), категорії, бюджети та теги. Схема побудована за принципами реляційних баз даних з нормалізацією до 3NF.

## Опис таблиць
### users
Зберігає основну інформацію про користувачів системи. Містить email, ім’я, пароль (у вигляді хешу) та дату створення.

Основні поля: `id`, `email`, `password_hash`, `name`, `created_at`

### user_settings
Зберігає індивідуальні налаштування користувача (валюта, мова, тема). Має зв’язок 1:1 з таблицею users.

Основні поля: `id`, `user_id`, `currency`, `language`, `theme`

### categories
Категорії для групування витрат та доходів користувача (наприклад: їжа, транспорт). Кожна категорія належить конкретному користувачу.

Основні поля: `id`, `user_id`, `name`, `type`, `color`, `created_at`

### transactions
Зберігає всі фінансові операції (витрати або доходи). Пов’язана з користувачем та категорією.

Основні поля: `id`, `user_id`, `category_id`, `amount`, `description`, `date`, `created_at`

### budgets
Визначає ліміти витрат користувача (загальні або по категорії) за певний період.

Основні поля: `id`, `user_id`, `category_id`, `limit_amount`, `period`, `start_date`, `end_date`, `created_at`

### tags
Теги для додаткової класифікації операцій (наприклад: “важливо”, “подорож”).

Основні поля: `id`, `name`, `created_at`

### expense_tags
Проміжна таблиця для реалізації зв’язку багато-до-багатьох між transactions і tags.

Основні поля: `expense_id`, `tag_id`

## Ключові зв’язки
- users → categories (1:N) — користувач має багато категорій  
- users → transactions (1:N) — користувач має багато операцій  
- users → budgets (1:N) — користувач має багато бюджетів  
- users → user_settings (1:1) — один користувач має одні налаштування  
- categories → transactions (1:N) — категорія використовується в операціях  
- transactions ↔ tags (M:N) — операції можуть мати кілька тегів
